# _scripts - Framework-Level Hard Constraint Scripts

This directory contains framework-level hard constraint validation scripts. These scripts provide deterministic validation when LLM executes certain critical operations, compensating for errors that may be caused by LLM randomness.

## Design Philosophy

```
┌─────────────────────────────────────────────────┐
│                    LLM Layer                    │
│   (Soft constraints: prompt rules, may be       │
│    violated due to randomness)                  │
└────────────────────┬────────────────────────────┘
                     │ calls
                     ▼
┌─────────────────────────────────────────────────┐
│                  Script Layer                   │
│   (Hard constraints: Python scripts,            │
│    deterministic execution)                     │
│   - Validate output format                      │
│   - Check required fields                       │
│   - Return errors for LLM to retry              │
└─────────────────────────────────────────────────┘
```

- **Soft Constraints (LLM Layer)**: Guide LLM behavior through prompt rules, but cannot guarantee 100% compliance
- **Hard Constraints (Script Layer)**: Validate output through deterministic scripts, return clear error messages for LLM to correct

## Difference from ai_workspace/

| Directory | Nature | Creator | Lifecycle | Git Tracked |
|:--|:--|:--|:--|:--|
| `_scripts/` | Framework infrastructure | Human | Persistent | ✅ Yes |
| `ai_workspace/` | LLM temporary workspace | LLM | Temporary | ❌ No |

- Scripts in `_scripts/` are part of the framework and should be version controlled
- Files in `ai_workspace/` are temporary LLM work products and can be cleaned at any time

## Script List

### validate_topic_tree.py

**Purpose**: Validate whether topic tree updates comply with hard constraint rules

**Validation Rules**:
- **R1**: All historical topic IDs must be preserved (prevent topic loss)
- **R2**: Prohibit using "omitted"/"..." markers to simplify history
- **R3**: Must contain "Last Updated" field
- **R4**: Status values must be one of predefined values (warning level)

**Usage**:

```bash
python _scripts/validate_topic_tree.py \
  --old ai_workspace/temp/discussion_topics.md.bak \
  --new ai_workspace/temp/new_topic_tree.md
```

**Output Format** (JSON):

```json
{
  "valid": true,
  "errors": [],
  "warnings": []
}
```

Or on failure:

```json
{
  "valid": false,
  "errors": ["R1 violation: The following topic IDs are missing: ['A', 'B']"],
  "warnings": []
}
```

**Exit Codes**:
- `0` - Validation passed (valid=true)
- `1` - Validation failed (valid=false)

### cleanup_topic_tree_temp.py

**Purpose**: Clean up temporary files generated by topic tree validation process

**Safety Mechanisms**:
- **Whitelist restriction**: Can only delete predefined filenames (hardcoded)
- **Path restriction**: Can only operate on `ai_workspace/temp/` directory
- **No arbitrary path exposure**: LLM cannot delete other files through parameters

**Usage**:

```bash
# Basic usage: clean up topic tree temp files
python cursor-agent-team/_scripts/cleanup_topic_tree_temp.py

# Preview mode: only show files to be deleted, don't actually delete
python cursor-agent-team/_scripts/cleanup_topic_tree_temp.py --dry-run

# Extended cleanup: include all .bak and .tmp files
python cursor-agent-team/_scripts/cleanup_topic_tree_temp.py --all

# Silent mode: no terminal output (still writes to log)
python cursor-agent-team/_scripts/cleanup_topic_tree_temp.py --quiet
```

**Parameters**:

| Parameter | Description |
|:--|:--|
| `--dry-run` | Preview mode, only show files to be deleted |
| `--all` | Extended cleanup, includes *.bak and *.tmp |
| `--quiet` | Silent mode, no terminal output |

**Output Format** (JSON):

```json
{
  "success": true,
  "deleted": ["discussion_topics.md.bak"],
  "skipped": [],
  "dry_run": false,
  "log_file": "ai_workspace/temp/cleanup.log"
}
```

**Log Location**: `ai_workspace/temp/cleanup.log`

**Exit Codes**:
- `0` - Success
- `1` - Failure

### cleanup_ai_workspace.py

**Purpose**: Safely delete files within the `ai_workspace/` directory

This script allows agents to freely delete temporary files in the workspace while ensuring no files outside the directory can be deleted. Solves the issue of Cursor requiring manual authorization for delete operations, preventing interruption of agent automation workflows.

**Safety Mechanisms**:
- **Hardcoded path**: Can only operate on `ai_workspace/` directory
- **Path validation**: Prevents path escape attacks (e.g., `../../../etc/passwd`)
- **Protected files**: Critical files (README.md, discussion_topics.md, etc.) cannot be deleted by default
- **Logging**: All operations are logged to `ai_workspace/temp/cleanup.log`

**Usage**:

```bash
# Delete a specific file (relative to ai_workspace/)
python cursor-agent-team/_scripts/cleanup_ai_workspace.py --file temp/old_note.md

# Delete a directory (recursive)
python cursor-agent-team/_scripts/cleanup_ai_workspace.py --dir temp/test_cleanup

# Delete by pattern (e.g., *.bak, *.tmp)
python cursor-agent-team/_scripts/cleanup_ai_workspace.py --pattern "*.bak"

# Delete files older than N days
python cursor-agent-team/_scripts/cleanup_ai_workspace.py --older-than 7

# Preview mode: only show files to be deleted, don't actually delete
python cursor-agent-team/_scripts/cleanup_ai_workspace.py --dry-run --pattern "*.tmp"

# Silent mode: no terminal output (still writes to log)
python cursor-agent-team/_scripts/cleanup_ai_workspace.py --quiet --file temp/old.md

# Force delete (including protected files, use with caution!)
python cursor-agent-team/_scripts/cleanup_ai_workspace.py --file README.md --force
```

**Parameters**:

| Parameter | Description |
|:--|:--|
| `--file <path>` | Delete specific file (relative to ai_workspace/) |
| `--dir <path>` | Delete directory (recursive) |
| `--pattern <glob>` | Delete by glob pattern (e.g., *.bak) |
| `--older-than <days>` | Delete files older than N days |
| `--dry-run` | Preview mode, only show files to be deleted |
| `--quiet` | Silent mode, no terminal output |
| `--force` | Force delete (including protected files) |

**Protected Files** (cannot be deleted by default):
- `README.md`
- `crew/README.md`
- `plans/README.md`
- `plans/INDEX.md`
- `prompt_engineer/README.md`
- `discussion_topics.md`

**Output Format** (JSON):

```json
{
  "success": true,
  "deleted": ["temp/old_note.md"],
  "skipped": [],
  "protected": [],
  "errors": [],
  "dry_run": false,
  "log_file": "ai_workspace/temp/cleanup.log"
}
```

**Exit Codes**:
- `0` - Success
- `1` - Complete failure (parameter error, permission issues, etc.)
- `2` - Partial failure (some files failed to delete, but some succeeded)

**Agent Usage Guide**:

Agents can call this script directly when cleaning up the workspace:
1. Clean up temporary analysis files: `--file` or `--pattern`
2. Clean up old files: `--older-than 7` (keep last 7 days)
3. Clean up test directories: `--dir temp/test_xxx`
4. Use `--dry-run` first to preview, then delete after confirmation

**Note**: Protected files can only be deleted with `--force`, use with caution.

### tts_speak.py

**Purpose**: Call macOS `say` command to read text aloud (TTS text-to-speech)

**Design Principles**:
- **Single responsibility**: Only calls the `say` command, no text preprocessing
- **Text preparation is agent's responsibility**: Agent must convert content to speakable natural language before calling
- **Framework-level universal feature**: All roles (/discuss, /crew, /prompt_engineer) can call
- **Disabled by default**: Only activated when user explicitly requests voice ("read to me", "speak it", etc.)

**Usage**:

```bash
# Basic usage
python cursor-agent-team/_scripts/tts_speak.py "Text to read aloud"

# Specify voice
python cursor-agent-team/_scripts/tts_speak.py --voice "Lili (Premium)" "High quality voice"

# Adjust speech rate (default 200 words/minute)
python cursor-agent-team/_scripts/tts_speak.py --rate 150 "Slow reading"

# Read from stdin (for long text)
echo "Long text content" | python cursor-agent-team/_scripts/tts_speak.py --stdin

# List available Chinese voices
python cursor-agent-team/_scripts/tts_speak.py --list-voices

# Check TTS environment
python cursor-agent-team/_scripts/tts_speak.py --check

# Force recheck (ignore cache)
python cursor-agent-team/_scripts/tts_speak.py --force-check
```

**Parameters**:

| Parameter | Description |
|:--|:--|
| `text` | Text to read aloud (positional argument) |
| `--voice`, `-v` | Voice name (default: Tingting) |
| `--rate`, `-r` | Speech rate, words/minute (default: 200) |
| `--stdin` | Read text from stdin |
| `--list-voices` | List available Chinese voices |
| `--check` | Check TTS environment and cache result |
| `--force-check` | Force recheck (ignore cache) |

**Environment Check**:

Automatically checks environment on first call, result cached to `ai_workspace/.tts_capability.json`.

Check items:
- Is operating system macOS
- Does `say` command exist
- Are Chinese voices available
- Can it read aloud normally

If check fails, script exits silently (exit code 3), no error message.

**Output Format** (JSON):

```json
{
  "success": true,
  "text_length": 15,
  "voice": "Tingting",
  "rate": 200
}
```

**Exit Codes**:
- `0` - Success
- `1` - Parameter error or execution failure
- `2` - System not supported (not macOS)
- `3` - TTS unavailable (environment check failed, silent exit)

**Agent Usage Guide**:

Before calling this script, agent needs to:
1. Confirm user explicitly requested voice output
2. Convert content to speakable natural language:
   - Tables → "This is a table with N columns, the first row is..."
   - Code → "This code does..."
   - Formulas → "This formula represents x equals y squared"
   - Images → "Here is an image showing..."
3. Remove all Markdown formatting symbols
4. Generate plain text then call the script

## Double-Buffer Validation Flow

LLM should follow this flow when updating the topic tree:

1. **Backup**: Copy current topic tree to `ai_workspace/temp/discussion_topics.md.bak`
2. **Generate**: Write new content to `ai_workspace/temp/new_topic_tree.md`
3. **Validate**: Call validation script to check new content
4. **Handle result**:
   - Validation passed → Overwrite official file with new content, delete temp files
   - Validation failed → Correct based on errors, retry validation
   - 3 consecutive failures → Restore backup, report error to user

## Adding New Scripts

When adding new hard constraint scripts, follow these guidelines:

1. **Output format**: Use JSON format uniformly, include `valid`, `errors`, `warnings` fields
2. **Exit codes**: Return 0 for success, 1 for failure
3. **Error messages**: Clearly describe the specific rule violated and content
4. **Documentation update**: Add script documentation to this README
